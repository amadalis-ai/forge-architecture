PREAMBLE
What This Document Is
This is not a product spec. This is not a feature list. This is not a pitch deck dressed up as prose.
This document is an architectural thesis. It makes a claim about the shape of the future — that autonomous AI will follow the same trajectory that all of computing has followed for seventy years: from interpretation to compilation. And it presents the architecture that proves the claim is not theoretical.
The thesis is this: the entire agentic AI industry has built its infrastructure on the interpreter pattern — systems that read an instruction, execute it, observe what happens, and decide what to do next, one step at a time, in an open loop. This works for demos. It works for simple tasks. It hits an architectural ceiling the moment the work becomes serious, the stakes become real, and the organization needs to trust what the AI produced.
The answer is not a smarter model. The answer is a better architecture around the model. An architecture that separates analysis from execution. That resolves ambiguity before runtime. That validates the coherence of a plan before the first byte of code runs. That produces deterministic, auditable, replayable execution from natural language intent.
The answer, in a word, is a compiler.
Forge is the proof that this works. It is a running system — not a roadmap, not a research paper, not a prototype. It compiles human intent into governed autonomous execution today, in production, with full audit trails, artifact checksums, and replay capability. This document explains why that architecture exists, how it works in precise technical detail, and why its implications extend far beyond one platform.
We are not building another AI platform. We are building the first compiler for a new kind of source code: human intent.


PART I
The Origin: A Compiler Written at Fifteen
I did not arrive at this from prompt engineering. I arrived at it from compilers.
I started programming as a child, moving through the early personal computing era — different operating systems, different memory models, different ways of thinking about what software even was. When I was fifteen, I built a Pascal compiler as a science fair project. Not a syntax highlighter. Not a formatter. A compiler — lexical analysis, parsing, symbol tables, intermediate representation, code generation. Later came a college thesis on compilers, then years building parser pipelines, type systems, and low-level execution machinery across enterprise systems. After that, decades of enterprise integration, data transformation, and systems architecture. The domains changed constantly. One instinct never left: separate analysis from execution. Resolve before you run. Validate before you emit.
When I encountered modern autonomous agents — the LangChains, the CrewAIs, the AutoGens — what struck me was not what they could do. It was what their architecture reminded me of.
It reminded me of interpreters.
The model reads a request. It decides what to do. It calls a tool. It sees the result. It decides again. Read, execute, hope, repeat. Every compiler engineer recognizes the failure shape immediately. Analysis and execution interleaved instead of separated. Problems discovered too late. No formal contracts between stages. No validation that the program is coherent before it starts running.
That was the moment the thesis clarified. The issue was not that agents needed to be smarter. The issue was that the architecture around them was too interpretive. And I knew — from thirty years of building systems — exactly what the alternative looked like.


PART II
How Everyone Else Builds AI — And Why It Hits a Ceiling
Before describing what Forge does differently, it is essential to understand the hidden assumption underneath every other agent platform built today. They all begin from the same premise:
The developer writes the orchestration code, and the model is called as a service inside that code. The application is primary. The model is a function that gets invoked. Intelligence is downstream of the scaffolding.
This is a natural mental model for software engineers. Write the program. Call the model when you need language capability. Wire the outputs. Handle the errors. The model is one component among many, forever subordinate to the application layer.
In practice, it looks like this: a developer builds an application in Python or TypeScript. Inside that application, there is a loop. The loop sends a message to the model, gets a response, decides whether the model wants to call a tool, calls the tool, sends the result back to the model, and repeats. The model proposes actions one at a time. The platform executes them one at a time. The model sees the result, proposes the next action.
This is the interpreter pattern — read, execute, read, execute — and every major agent platform uses it. LangChain. CrewAI. AutoGen (before Microsoft killed it). Dify. The new wave of so-called agent platforms that give you a UI to configure the code that calls the model. The architecture underneath is always the same.
It works. For simple tasks, it works well. But it hits a structural ceiling the moment the work gets serious:
Late failure discovery. If the model hallucinates a tool name on step twelve, you find out on step twelve — after steps one through eleven have already run and cost money. If step twenty needs an artifact from step three and the formats are incompatible, you discover this at step twenty. There is no pre-execution phase that catches this.
Context degradation. If the model accumulates context over a long execution session, its reasoning quality degrades. Every step carries the weight of every prior step's outputs, observations, errors, and side effects. By the time you reach step forty, the model is working through a fog of accumulated context that it was never designed to manage.
No formal contracts. There are no declared artifact types, no schema validation between steps, no checksums, no provenance tracking. Data flows forward on hope. If step five produces output that step eight cannot consume, the failure is discovered at step eight — after steps five through seven have already spent tokens and time.
These are not implementation bugs. They are architectural consequences of interleaving analysis with execution. And no amount of prompt engineering, tool improvement, or model scaling will fix them — because the problem is not the model’s capability. The problem is the shape of the system around it.
The Industry Knows This
To be clear: serious teams have been working on these problems. LangGraph introduced state graphs with checkpointing and structured handoffs. CrewAI added task delegation with expected outputs and dependency declarations. Google’s A2A protocol defines agent-to-agent communication standards. Prefect and Dagster have had artifact contracts and dependency management for years in the data pipeline world. OpenAI’s Agents SDK has structured handoffs. These are real efforts by talented engineers.
But they are all solving it at the application layer — adding patterns, conventions, and APIs on top of the interpreter architecture. The framework gives you tools to declare that Step 3 should produce output for Step 5. The developer is responsible for writing that declaration, maintaining it, and handling what happens when it breaks. The model can still hallucinate tool names unless you write code to prevent it. Governance is still a separate concern you wire in. The framework helps you build the plumbing — it does not eliminate the need for plumbing.
The distinction is not that other frameworks ignore these problems — many don’t. The distinction is where the solution lives. In application-layer solutions, correctness depends on the developer doing the right thing. In a compilation architecture, correctness is a structural property of the system.


PART III
The Fundamental Inversion
Forge inverts the relationship between the model and the code.
Other platforms: the developer’s code controls the AI. The developer writes the orchestration. The model is called as a service inside that code. The model responds when asked, does what it is told, and has no environment of its own.
Forge: the AI controls the code — inside a system that governs, validates, and verifies what it produces.
The model is not a service you call. The model is the mind. Everything else — the planning structure, the execution environment, the governance layer, the artifact contracts, the skill packages, the sandbox — is an operating system built around that mind to let it think, plan, write code, and produce real work.
We don’t just call the model as a service. We provide a welcoming and nurturing environment. An operating system and framework and structure around the model to allow the model to do what it does best: think. We don’t call the model. The model thinks. And we quote it.
This inversion changes the unit of design. Instead of writing orchestration code that invokes AI capability, the system is designed so that a human objective becomes source code, and the AI is the compiler that transforms it into governed, executable, auditable work.
That operating system includes the semantic layer that gives structure to intent, the multi-pass compiler that translates meaning into executable plans, the runtime policies that freeze governance before execution begins, the capability binding system that decides what the model may actually use, the execution substrates that carry different forms of work, the verification and audit systems that produce receipts not vibes, and the replay architecture that makes successful work reusable without re-spending a single token on re-planning.
The model is not a subroutine. It is the intelligence inside a compiler-driven operating system.


PART IV
The Model Thinks in Code
Before describing the architecture, it is worth saying something important about what the model is actually doing inside this system — because it is not what most people picture when they hear “AI agent.”
When Forge executes a step, the model does not produce a text answer. It does not describe what should happen. It does not call a tool and wait for a result. It thinks in code.
The model writes programs. It reasons through what needs to happen, produces executable code that implements that reasoning, runs the code inside the sandbox, observes the result, and continues. The code is not incidental output — it is the model’s reasoning made executable and verifiable.
Consider the difference. If the model says “the average revenue is $47,000,” you have no way to know whether that came from actual computation or from statistical prediction — the model predicting what a plausible revenue figure looks like based on its training data. But if the model wrote and ran ten lines of code that opened the data file, read every row, computed the mean, and printed the result — you can read those ten lines. The reasoning is the code. The answer came from computation, not prediction.
This is Code-Act: the model reasons by acting in code, not by predicting in prose.
The concept originates from the CodeAct research paper by Xingyao Wang et al. — a body of work that came out of Apple’s research group and demonstrated that models that act through code dramatically outperform models that act through tool-calling alone. When I read that paper, my mind could not have been clearer. The evidence was overwhelming, the architecture was elegant, and the question it left me with was not whether this was the right approach — it was why isn’t everyone already doing this? The answer, I believe, is that Code-Act demands an entirely different kind of infrastructure. You cannot just let a model write code and hope for the best. You need a governed sandbox, artifact contracts, verification pipelines, and a compilation architecture that makes the code trustworthy. That infrastructure is Forge.
Why Code-Act Changes Everything
Verifiability. Code produces deterministic outputs from deterministic inputs. You can audit what the model did by reading what it wrote. The programs are right there in the execution record. When a model tells you a number, you trust it. When a model shows you the code that computed the number, you verify it. That distinction — between trust and verification — is the gap that Code-Act closes.
Replayability. A program that was written and executed once can be executed again — with different data, on a schedule, without the model ever thinking about it again. Code is a durable artifact. A conversation is not.
Hallucination elimination. A model that has to write working code to produce an answer cannot fabricate that answer and get away with it. The code either runs and produces a result, or it fails and the system knows it failed. There is no plausible-sounding wrong answer that gets through undetected, because the system is not asking the model for an answer — it is watching the model compute an answer through verifiable programs.
This is not a small distinction. This is the difference between an AI that tells you things and an AI that does things and proves what it did.


PART V
The Plan: Not a List — A Compiled Dependency Graph
This section matters deeply, because it is where Forge diverges from every other system in the world — and where every other system trivializes what planning actually means.
When someone says “plan” in the context of AI agents, most people picture a numbered list. Step 1: do this. Step 2: do that. Step 3: produce the output. That is what ChatGPT gives you. That is what Cursor gives you. That is what Claude Code gives you. A markdown list of things to do.
That is not what Forge produces. Not even close.
When a human objective enters Forge, the first thing that happens is not execution. It is structural decomposition. A frontier reasoning model with extended thinking takes the objective — “audit these timesheets against the rate cards and tell me what to invoice” — and does something closer to what an architect does with a building specification than what a chatbot does with a question. It thinks — deeply, for as long as the reasoning model needs. Not the kind of instant response that comes from pattern prediction. Extended thinking — the model working through the problem internally, considering dependencies, edge cases, failure modes, and sequencing — before it produces any output. Forge does not control or limit this thinking time; it supports whatever reasoning budget the provider’s API allows and the user configures. The planning model thinks until it is done thinking.
Then it produces a plan. But the plan is a dependency graph. A program in its own right. Every step declares what it is trying to produce, what prior steps it depends on, what success looks like, what artifacts flow in and what artifacts flow out. The dependencies form a directed acyclic graph that can be validated for cycles, checked for completeness, analyzed for parallelism opportunities. The planner reasons about the shape of the entire operation before a single tool is called, a single file is read, or a single byte of code is written.
Our plan is not a markdown list. It is a graph — a program for itself. It can be validated. It can be compiled. It can be transformed from one representation into another. The complexity involved is extraordinary. This is not step one, step two. This is a formal structure with declared contracts at every edge.
This is where the expensive AI earns its cost. One minute of deep planning prevents twenty minutes of failed execution and wasted money downstream. The planning model is a frontier reasoning model — the most capable and most expensive. But it only thinks once. Every subsequent stage of the pipeline uses cheaper, faster models. You pay for thinking, not for failing.
Steps, Sub-Steps, and Surgical Batching
From the compiled plan, each step is decomposed further into sub-steps — the individual operations that together accomplish the step’s objective. This is where Code-Act comes alive, and where the architecture diverges most sharply from every interpreter-based system.
Here is what actually happens when a step executes.
The step executor — a fresh AI model with clean context — receives the step’s compiled contract: the objective, the declared inputs, the expected outputs, the allowed tools, the materialized skill packages, the success criteria. Nothing more. It does not inherit the conversational context of prior steps. It does not carry the weight of decisions made three steps ago. It starts clean, with exactly the information it needs.
The model then thinks through the entire step. It reasons about how to accomplish the objective. And then it writes all the code — the programs that implement the sub-steps — as a connected sequence. Sub-step one downloads the data. Sub-step two parses and normalizes it. Sub-step three computes the metrics. Sub-step four writes the output file. The code for each sub-step links to the next.
This is a critical distinction. The sub-steps for an entire step are compiled together and sent to the AI Executor inside the secure sandbox as a batch — not streamed one decision at a time. The model thinks through the entire step, writes all the code that links sub-step to sub-step, and hands the complete program to the execution engine. The executor runs it.
If it fails at sub-step three of seven, the failure is specific, localized, and surgical. The system knows exactly which sub-step failed, what the error was, and where to retry — not from the beginning, but from the point of failure. If sub-step nine of step five fails, repair starts at sub-step nine. Not from step one. Not from step five. From the exact point where the failure occurred. No wasted money re-running things that already worked.
This is the difference between planning and executing on the one hand, and executing and hoping on the other.


PART VI
The Five-Stage Compilation Pipeline
Different questions must be answered by different stages. Interleaving them is what produces interpreters. Separating them is what produces compilers.
Stage A — Structural Planning (Parsing and AST Generation)
The planner decomposes the mission into an abstract step graph: steps, dependencies, output paths, validations, and semantic input/output declarations. Each step declares what it is trying to produce, what it depends on, and what success looks like — in abstract semantic terms, not in tool IDs or file paths.
What the planner explicitly does not produce: tool IDs, skill IDs, concrete profile identifiers, adapter decisions, backend routing choices, or any code. The planner reasons about the problem structurally before it reaches for solutions. This constraint is enforced — it is not a guideline.
Stage B — Semantic Intent Resolution (Symbol Resolution)
A separate model pass determines, for each step, which domain pack and workflow pack best describe the work. These are not abstract labels. They are formal objects in the compiler’s type system — structured registries of entity schemas, contract profiles, canonical workflows, trigger patterns, and disambiguation rules that the compiler uses to resolve intent into typed, validated execution contracts.
A domain pack answers: what business family does this step belong to? Each domain pack is a structured object that contains the entity schemas relevant to that domain (billing.time_entry, billing.rate_card, billing.discrepancy_row), the canonical contract profiles that govern how artifacts are typed and validated (billing.table.json_rows.time_entries.v1), the default workflow sequences for that domain, trigger patterns that activate it ("invoice discrepancies", "timesheet reconciliation", "billable hours"), example user goals that help the resolver match intent, and explicit disambiguation notes that tell the compiler when to choose this domain over a similar one.
A workflow pack answers: what operation is being performed? Each workflow pack defines the operational semantics of a class of work. It declares what input and output slots it expects (e.g., the reconcile-compare workflow expects normalized_rows or joined_rows as input and produces findings, scores, or report_artifact as output), what canonical profiles govern those slots, what trigger patterns activate it ("compare two sources", "find discrepancies", "cross-reference and flag mismatches"), and disambiguation notes that distinguish it from superficially similar workflows.
The resolver does not guess. It matches the step’s objective against the trigger patterns, synonym vocabularies, and example goals in the domain and workflow registries. When it determines that a step belongs to billing-audit with a reconcile-compare workflow, it has resolved not just a label but an entire semantic context: the entity schemas in play, the contract profiles that will govern handoffs, the validation rules that apply, and the adapters that are available. This classification flows forward into Stage C, where the compiler uses it to bind concrete profiles and enforce compatibility.
The system currently includes domain packs for research-synthesis, investigation-core, resume-audit, billing-audit, finance-analysis, legal-review, website-build, code-modification, data-merge-transform, sales-crm, PSA-capacity, and security-audit — with workflow packs for file-acquisition, file-batch-load, ingest-materialize, normalize-transform, reconcile-compare, analyze-score-rank, research-fetch-summarize, render-report, render-website, code-modification, deploy-publish, and investigation-parallel. Each one is a rich structured object, not a keyword. The full anatomy of these objects is detailed in Part VIII.
Stage C — Contract Compilation (Type Checking and Linking)
The compiler links the abstract plan against the registry. For every input and output slot across every step, it resolves concrete profiles, enforces producer/consumer compatibility, inserts explicit adapters where representation families differ, and fails closed where compatibility cannot be established.
The output is a set of frozen step contracts. Each one declares what artifact arrives at each input slot, what schema it must have, where it came from, what validations must pass, and what a downstream step may assume. These contracts are not advisory. They are the execution boundary conditions.
Compatibility rules are explicit: same semantics and representation means direct match. Same semantics but different representation means adapter required. Finer to coarser grain means aggregation adapter. Coarser to finer means fail closed. Missing required fields means incompatible. No silent coercion. Never “close enough.”
Stage D — Backend and Capability Resolution (Code Generation)
Different work belongs in different execution substrates. The planner emits runtime traits per step — not backend choices, but requirements: code execution needed or not, preferred language family, shell access needed, native binaries, large working sets, external network, preview or deployment behavior.
A backend resolver maps those traits to the right substrate: the tool loop for lightweight bounded work, edge isolates for fast typed JavaScript/TypeScript execution and report assembly, sandboxed containers for Python-heavy analysis and shell workflows and OCR and document conversion, and the hosted application lane for long-lived preview deployments.
Backend choice is compiled, not improvised. And crucially, different steps in the same plan can run in different substrates, on different models. This is intentional and deeply important.
Stage E — Isolated Execution (Runtime)
Each step executes as a fresh mind inside a governed environment. The step executor receives the step objective, the compiled contract, the resolved capabilities, the allowed tools, the materialized skill packages, and the exact artifacts it is permitted to read and write. Nothing more.
It does not inherit the conversational context of prior steps. It does not need to remember what happened three steps ago. It does not guess what a filename means — the contract tells it. It does not choose between Python and JavaScript from intuition — the backend resolver already decided.
The system wraps the model’s work in system-owned phases: workspace preparation, artifact materialization, preflight receipt generation, the actual execution, postflight verification, output persistence. The model cannot skip verification. It cannot bypass persistence. It cannot produce untracked outputs.


PART VII
Each Step Gets a Fresh Mind
This is a subtle but critical architectural decision that separates Forge from every interpreter-based platform.
In an interpreter-based system, the model carries a growing context window. By step twenty, it is working through every command it issued, every output it observed, every error it encountered, and every decision it made across the entire session. Even with context compression techniques, there is quality degradation over long sessions. The model’s reasoning becomes less precise as it struggles to maintain coherence across an ever-expanding history.
Forge eliminates this entirely. Step twenty is as sharp as step one — because it is step one, for that agent. The step executor receives only its compiled contract, its verified inputs from the previous step, and its materialized capabilities. It starts clean, does its work, produces its outputs, and is done.
Between every step, two things happen. First, the platform mechanically verifies that the expected output files exist and their checksums match. Second, a separate AI model — not the same model that produced the work — reviews the actual content. Is the file empty? If it is HTML, does it actually contain markup? If it is a data analysis, do the numbers look plausible? If anything looks wrong, it is flagged before the next step starts. Bad output never cascades forward.
The model does not get to mark its own homework.


PART VIII
The Semantic Layer: A Type System for Autonomous Work
The compiler needs a type system. In traditional compilers, that type system is formal syntax — integers, strings, structs, function signatures. In an autonomous work compiler, it is a family of semantic objects that give structure to natural language intent.
This is not a metaphor. These are real, structured objects in the system — a formal object model that the compiler resolves against, validates against, and enforces at every step boundary. What follows is the actual anatomy of the type system — not a summary, but the real objects, the way the compiler sees them.
Artifact Kinds: How Data Is Shaped
An artifact kind declares the representation format of a piece of data — its shape, not its meaning. Kinds use a hierarchical key notation that classifies artifacts by family and serialization.
FamilyExample Kind KeysWhat It DescribesFilesfile.binary file.text file.batch_manifestRaw file artifacts — opaque binaries, plain text, or manifests describing a batch of files to processTablestable.csv table.json_rows table.parquetTabular data in various serializations. The kind tells you the format; the entity schema tells you what the rows meanDatasetsdataset.joined_rows dataset.aggregated_rowsProcessed data: rows joined across sources, or rows grouped and summarizedReportsreport.markdown report.html report.pdfHuman-readable deliverables in different output formatsWebweb.page.html web.bundle.staticWebsite artifacts: single pages or deployable static bundlesCodecode.patch.unified_diff code.bundle.projectSource code artifacts: unified diffs for surgical edits, or full project snapshotsMetadatadeployment.receipt validation.results evidence.json_rowsLifecycle and provenance artifacts: deployment receipts, QA results, evidence rows
A kind tells the compiler what format an artifact is in. It does not tell the compiler what the artifact means. That is the job of entity schemas.
Entity Schemas: What Data Means
An entity schema declares the semantic identity of data — what the rows or objects actually represent. A file with kind table.json_rows tells you the format. The entity schema tells you whether those rows are billing time entries, legal clause findings, or resume audit scores. That distinction must be declared, not inferred from a filename.
Each entity schema declares its required fields. Here is what three different entity schemas look like — same kind, completely different meaning:
billing.time_entry
Required Fieldstime_entry_id, employee_id, project_id, contract_id, customer_id, service_date, hours_decimal, billable_flag, work_type, source_artifact_idsRepresentsOne billable work entry: who worked, on what project, for how long, at what date, under which contract. The atomic unit of billing reconciliation.legal.clause_finding
Required Fieldsdocument_id, clause_id, clause_type, party_a, party_b, effective_date, termination_date, risk_level, citation_ids, evidence_card_ids, source_artifact_idsRepresentsOne finding from a legal clause analysis: which document, which clause, what type (indemnity, termination, liability cap), the parties, the risk level, and traceable citations back to the source text.resume.resume_audit_row
Required Fieldscandidate_id, job_requirement_id, score, strengths, risks, recommendation, citation_ids, source_artifact_idsRepresentsOne candidate evaluated against one job requirement: the score, identified strengths and risks, the recommendation, and citations back to the resume content that supports the evaluation.
Notice that source_artifact_ids appears in every entity schema. This is how the system maintains traceability: every piece of data knows which upstream artifact produced it. Under a provenance-required policy profile, missing provenance blocks execution. This is not optional decoration. It is a structural requirement of the type system.


Contract Profiles: The Full Type Signature
A contract profile is where kind and entity schema come together with enforcement rules. It is the complete type signature of an artifact — the format, the meaning, the required fields, the grain, and the provenance requirements — all in one declaration. Here is a real contract profile, the way the compiler sees it:
billing.table.json_rows.time_entries.v1
Kindtable.json_rows — a JSON array of row objectsEntity Schemabilling.time_entry — each row is a billable time entryTop-Level Typearray — the file must parse as a JSON arrayRequired Fieldstime_entry_id, employee_id, project_id, service_date, hours_decimal, billable_flag, source_artifact_idsRow Graintime_entry — each row is one entry, not an aggregationAdditional FieldsAllowed — schema is open for domain extensionsProvenancesource_artifact_ids required — every row traces to its source
When the reconcile-compare step produces discrepancy rows, those are validated against a different profile — billing.table.json_rows.discrepancy_rows.v1 — with a different entity schema, different required fields (discrepancy_id, discrepancy_type, discrepancy_reason), and a different grain (discrepancy). If a required field is missing, the step fails with a specific diagnostic. The contract was violated.


Domain Packs: The Standard Libraries
A domain pack is the richest object in the type system. It assembles everything the compiler needs to understand a business domain: the entity schemas, the contract profiles, the default workflows, the trigger vocabulary, and the disambiguation rules. Here is the anatomy of a real domain pack:
Domain Pack: billing-audit
PurposeReconcile time, rates, contracts, or invoice lines. Determine what should be billed, what was billed incorrectly, or where discrepancies exist.Entity Schemasbilling.time_entry billing.rate_card billing.invoice_line_item billing.discrepancy_row billing.billing_summaryContract Profilesbilling.table.json_rows.time_entries.v1 billing.table.json_rows.rate_cards.v1 billing.table.json_rows.invoice_line_items.v1 billing.table.json_rows.discrepancy_rows.v1 billing.report.html.audit_summary.v1Default Workflowsfile-acquisition ? normalize-transform ? reconcile-compare ? render-reportTrigger Patterns"billing audit", "invoice discrepancies", "timesheet reconciliation", "what to invoice", "billable hours"Synonym Vocabularytimesheets, time entries, hours logged, billable time, T&M, rate cards, fee schedules, bill rates, reconciliation, discrepancies, variances, SOWDisambiguationUse billing-audit when the mission involves matching time, rates, contracts, or invoice lines. Use finance-analysis when the mission is about ledger entries, budgeting, or financial performance. Use data-merge-transform when the task is only structural reshaping without billing semantics.
The legal-review domain pack follows the same structure but carries entirely different semantics:
Domain Pack: legal-review
PurposeAnalyze contracts, policies, legal text, obligations, liabilities, or clause-level risk. Same structure. Completely different semantics. The compiler does not care what the domain is about. It cares that the types resolve, the profiles are compatible, and the contracts are satisfied. Twelve domain packs exist today: research-synthesis, investigation-core, resume-audit, billing-audit, finance-analysis, legal-review, website-build, code-modification, data-merge-transform, sales-crm, PSA-capacity, and security-audit.


Workflow Packs: The Shape of Operations
A workflow pack captures the operational semantics of a class of work — what goes in, what comes out, and how the compiler validates the transition.
Workflow Pack: reconcile-compare
PurposeCompare two or more compatible inputs to identify matches, mismatches, discrepancies, or missing records.Input Slotsnormalized_rows, joined_rowsOutput Slotsfindings, scores, report_artifactCanonical Profilesbilling.table.json_rows.discrepancy_rows.v1 finance.table.json_rows.variance_rows.v1Trigger Patterns"compare two sources", "find discrepancies", "reconcile datasets", "cross-reference and flag mismatches"Disambiguationreconcile-compare requires two or more input sources. analyze-score-rank operates on one source and scores it. normalize-transform emits reshaped data without comparison semantics.
The disambiguation rows encode the kind of knowledge a senior architect carries in their head — except formalized, versioned, and machine-readable. They prevent the compiler from confusing reconcile-compare with normalize-transform even when a user’s natural language could plausibly match either.
Policy Profiles and Adapters
Policy profiles are governance rules applied at compile time, not runtime. The strict.fail-closed.v1 policy requires all consumer-required fields, allows no implicit normalization, blocks on any mismatch, and enforces provenance exactly. The strict.allow-normalizers.v1 policy permits declared adapters but still blocks on unresolvable mismatches. The provenance.required.v1 policy blocks completion if source artifact IDs are missing. A plan compiled under strict.fail-closed.v1 will reject adapter insertion that the same plan under strict.allow-normalizers.v1 would accept. The governance is structural.
Adapters make representation differences explicit. When the compiler encounters a producer that outputs generic normalized rows and a consumer that expects billing time entries, it checks the policy. If adapters are allowed, it finds the declared adapter (adapter.generic.normalized-rows.__to__.billing.table.json_rows.time_entries.v1) and inserts it as an explicit, auditable step in the graph. If the policy is fail-closed, it rejects the plan. There is no “close enough.”
The Evolution of the Compiler Language
This is one of the most important things to understand about Forge, and one of the most exciting.
The compiler works today. Plans are generated, compiled, and executed successfully. But the compiler’s language — the domain packs, workflow packs, entity schemas, contract profiles, adapters — is at the very beginning of its evolution. Think of it the way you would think about the earliest days of programming languages. The compiler exists. The foundational constructs work. But the language features are evolving with each new domain, each new use case, each new class of work.
And here is why that is transcendent, not limiting:
Every domain becomes a compiler target. When we build a domain pack for legal work — with entity schemas for clause findings and obligation rows, with contract profiles that enforce required fields and provenance, with workflow packs that define the sequence from ingestion through analysis to risk-register rendering, with trigger patterns and disambiguation rules — we are not building an agent. We are not writing a prompt. We are building a legal compiler. The model programs against a rich semantic environment the same way a developer programs against a standard library.
Today the system has twelve domain packs and twelve workflow packs with dozens of contract profiles, entity schemas, and adapters. Tomorrow it will have hundreds. Each one makes the compiler smarter, more precise, and more capable of validating work in a new domain — without changing a line of the compiler itself. The language grows. The compiler stays.
This is not building agents. This is building a standard library for autonomous work — one domain at a time. And the compiler ensures that every program written against that library is valid before it runs.


PART IX
The Replay Architecture: Work That Compounds
This is where Forge departs most sharply from everything else built to date — and where it delivers the most value to enterprises.
When a plan executes successfully, the result is not just output files. The result is a Capsule — a complete, self-contained, reusable execution unit that contains the compiled step graph, the code the model wrote for each step and sub-step, the input and output manifests, the skill bindings, the validation results, and the exact artifact contracts that governed the run.
That Capsule can be replayed. Without re-planning. Without re-spending on a frontier reasoning model. Without re-discovering capabilities. Without re-generating code. The compilation already happened. The programs already exist. They are deterministic. They can be executed again.
Sealed Capsules: Fully Deterministic Execution
Some work is structurally repetitive. The same contract arrives, with different data, on a schedule. Convert this week’s timesheet exports to canonical billing format. Download the API snapshot and compute the enriched metrics. Transform inbound records into the normalized schema.
For this class of work, the sealed Capsule runs completely — without any model inference at all. The executor runs the compiled programs on the new data. The validator checks the outputs. The results are persisted. No tokens spent on planning. No tokens spent on code generation. No tokens spent on capability discovery.
An operation that cost a full planning run the first time costs almost nothing on every subsequent execution. The intelligence was amortized at compile time. The code was proven. The contracts were validated. Now it just runs.
Think about what this means. A non-technical user — not a developer, not an engineer — described what they wanted in natural language. The system compiled it, executed it, and proved it worked. Then they pressed a button to encapsulate it. Now it runs reliably, repeatably, governed by the same contracts, for a fraction of the cost. This person just wrote a script. Without knowing how to code. Without configuring a workflow builder. They described an objective, and the compiler produced a reusable program from it.
Structural Capsules: Fresh Reasoning Within a Frozen Architecture
Some work requires judgment that depends on content. Analyzing a resume requires part code — parsing, field extraction, scoring formula computation — but it also requires the model to reason over the actual substance of the resume, weigh qualifications against criteria, and make decisions that cannot be pre-programmed.
For this class of work, the structural Capsule preserves the architecture but opens it for fresh reasoning. The compiled step graph is preserved — the structure, the contracts, the sub-step decomposition, the validation requirements, the model assignments. But the generated code for each step is stripped. The model executes fresh within the frozen architecture, reasoning over the new content, writing new sub-step code as needed, producing outputs that conform to the same contracts.
The result: you get the structure of a carefully compiled plan without paying to generate that structure again, and you get fresh model reasoning over new content without the model having to figure out how to approach the problem from scratch.
Everything that was deterministic stays deterministic. Everything that requires thinking gets fresh thinking. And you did not pay for a new plan. This is a workflow on steroids that anybody can create via natural language.
The Economics of Capsules
If you have a process that reliably audits five hundred resumes — one that has been proven to correctly ingest files, extract structured data, apply scoring criteria, produce ranked outputs, and generate a report — you can encapsulate that entire execution run. The file ingestion works exactly as it did before. The data normalization follows the same contracts. The scoring formulas execute the same code. The only thing that changes is that the model gets fresh reasoning over the content of each new resume — because that is the part that requires judgment.
You can also govern what model runs each step. The data download might use a fast, cheap model. The normalization uses the same cheap model. The creative analysis routes to a frontier reasoning model. The report generation uses a model known for creative quality. These assignments are compiled into the Capsule. They are not improvised at runtime.
And Capsules are designed for headless execution. The entire lifecycle is API-driven. Any system that can make HTTP requests can dispatch a Capsule run — no UI required. This is the foundation for trigger-based automation: an external event fires, a webhook receives it, a Capsule run is dispatched with parameters extracted from the event payload. Deterministic, governed, auditable automation — triggered by events, executed by proven compiled plans.
Imagine the cost savings alone. Imagine the reliability.


PART X
Specialist Workspaces: An Agent Is Not a Prompt
Let me state this clearly: today, everybody thinks an agent is just a prompt. You write a prompt, you let it use some tools, and you call it an agent. That is not an agent. That is just a set of instructions for a model.
Forge lives in a different plane. Because the model thinks in code — following Code-Act — and because we can tweak every single phase of the plan generation, compilation, and execution pipeline, we enable the creation of something far more powerful than an agent: Specialist Workspaces.
A workspace in Forge is not a chat window. It is a persistent environment — an office where the AI has files, memory, tools, and rules, and the user interacts through natural language conversation. Each workspace has its own file system, its own memory, its own governance policies, its own knowledge base, its own model assignments, its own skill configurations.
When an enterprise needs to create a workspace for their accounting team, or their legal research division, or their compliance department, they are not just changing a prompt. They are creating a dedicated computing environment. The system prompts, the specific skills, the allowed tools, the egress rules, the knowledge bases, the RAG configurations, the reasoning levels, the timeouts, the model assignments for each phase of the pipeline — everything is refined and specifically configured for that profile.
You are not building an agent. You are building a specialist. The entire workspace from knowledge bases to file systems to RAG to the prompts to the instructions to the sandbox is tailored and created for that particular function. This is a mixture of specialists — the same principle that makes modern models powerful, applied at the platform level.
Deep Configurability

The model assignments per phase — which AI model handles planning, which handles execution, which handles auditing.

PART XI
Per-Step Model Governance
This is one of Forge’s most powerful and least obvious architectural decisions.
Different steps in the same plan can be governed by different AI models. Not as a theoretical possibility — as a compiled, configured, enforceable policy.
Consider a plan that downloads data from an API, normalizes it into a canonical format, performs a creative analysis, and generates an HTML report. The data download step is mechanical — it needs speed and cost-efficiency, not creativity. The normalization step is deterministic — it follows a schema. The creative analysis step requires judgment and nuance. The report generation step needs strong writing and design sensibility.
In Forge, the data download routes to a fast, cost-efficient model. The normalization uses the same cheap model. The creative analysis routes to a frontier model with strong reasoning. The report generation routes to a model known for creative quality. Each assignment is compiled into the plan. It is not decided at runtime. It is not the same model forced to do everything.
Model selection is model-agnostic. Forge is not locked to any provider. The planner can be an OpenAI reasoning model. The executor can be Claude for creative work or a lightweight model for routine data processing. The auditor can be a precision-focused model from any provider. Different models from different providers, in the same plan, governed by configuration — not code changes.
This is how Forge achieves its cost structure. The expensive frontier reasoning model thinks once, during planning. Every execution step can use cheaper, faster models that follow the compiled contracts. The total cost of the compilation approach is lower than the interpreter approach, because you pay for thinking instead of paying for failing and retrying.


PART XII
Skills as Linked Libraries
Skills in Forge are not prompt templates. They are versioned capability packages.
When a step’s compilation determines it needs a specific skill — say, the HTML report builder — the entire package is materialized into the sandbox: behavioral instructions, technical reference, executable scripts for rendering, validation, and conversion, CSS, templates, chart libraries, and schema definitions. The model calls the rendering script as a subprocess. It does not need to know how to build HTML reports from scratch. It calls the rendering pipeline with the right inputs and gets verified output.
This is library linking. The skill is a pre-compiled, versioned, tested capability module that loads at step initialization and is released when the step completes. The context window stays focused. The model gets exactly the capabilities it needs for this step, and nothing more.
Skill governance is enforced at compilation time, not suggested at runtime. Execution profiles freeze a skill policy: required skills that must be present, preferred skills that load if available, denied skills that cannot appear in any plan regardless of what the model wants. Governance cannot drift mid-execution. Audit consistency is guaranteed by immutability, not by logging.
Skill discovery is system-driven, not model-driven. When the planner declares that a step needs data analysis capability, the system searches the available skills via vector semantic search against the step’s objective. The model never browses a flat list and picks what looks right. The system discovers what is available, the model selects from what was discovered. If a capability does not exist in the registry, it cannot be hallucinated into the plan.


PART XIII
Why Enterprises Care
Enterprises do not buy autonomy because it looks magical. They buy it if it can be trusted — and if it compounds.
Trust and auditability. Every step has preflight receipts, postflight receipts, SHA256 checksums, execution evidence, and a contract hash. The entire compiled plan is a durable, inspectable artifact. An auditor can trace from user intent through compiled plan through step execution through artifact production. There are no gaps. The model does not get to tell the system the work is done — the operating system independently verifies that contracts were fulfilled.
Cost control. The interpreter pattern wastes money on unnecessary retries, repeated context stuffing, duplicate tool discovery, wrong-runtime execution, and late discovery of incompatibility. The compiler eliminates most of that waste by resolving before running. The replay architecture eliminates the rest by amortizing the planning cost across every subsequent execution.
Governance. The system must know what policy applies, when provenance is required, when human review gates apply, what side effects are permitted, what backend is lawful for the task. These are compiler-era questions. They are answered during compilation, frozen at run creation, and enforced structurally — not by instructing the model to follow rules, but by making the rules the boundary conditions of the execution environment.
Compounding intelligence. Every successful execution builds a Capsule. That Capsule is organizational knowledge — structured, validated, replayable. The enterprise is not just buying AI runs. It is accumulating a library of compiled autonomous work that gets cheaper to execute over time. The first billing reconciliation costs a full planning run. The hundredth costs almost nothing. The knowledge compounds.
Budget enforcement. Cost caps are checked after each model call. Execution halts before the cap is exceeded — not after.
Data security. All processing happens within governed environments. Egress rules control what external sites agents can reach. No API keys in agent code. Multi-tenant isolation means one organization’s data never touches another’s.


PART XIV
Why This Is Hard to Replicate
You cannot replicate this by adding a planner to a sandbox, adding a semantic registry to a tool loop, adding policies to a multi-agent chain, or adding validation to an existing pipeline.
The hard part is the composition, and the pieces are interdependent.
The semantic layer gives language-to-structure mapping. Without it, the compiler has no type system. The structural planner gives abstract decomposition. Without it, there is no separation of analysis from execution. The intent resolver assigns the right semantic worldview. Without it, the compiler cannot resolve the right contracts. The compiler freezes contracts. Without it, steps receive hints, not guarantees. The backend resolver picks the right substrate. Without it, execution is one-size-fits-all. The skill resolver bounds capability selection. Without it, the model can fabricate what it reaches for. The step executor acts inside a frozen world. Without it, each step inherits the noise of every step before it. The replay architecture captures the Capsule. Without it, every run pays full planning cost.
A skills registry without a sandbox to materialize skills into is a catalog, not a linker. A sandbox without artifact manifests can propagate files but cannot verify contracts. Execution profiles without policy freezing can enforce rules in the moment but cannot guarantee audit consistency over time. Replay without compiled contracts is just re-running code without knowing whether it is still correct.
These cannot be added incrementally to an interpreter. The interpreter has to become a compiler. And that is a rebuild.
The 18-stage validation pipeline (L0–L6) is the deepest data quality infrastructure in the agentic AI market. No competitor has anything resembling it. This pipeline runs automatically on data ingestion, tool invocation, API spec import, and step output validation. It is genuine, defensible intellectual property.

Structured languages gave way to richer type systems, compilers, linkers, and runtimes. Each step made human intent more writable and machine execution more reliable.
We are entering that same transition again. The source language is no longer Python or COBOL. It is human language.

Forge is the first system that treats the plan as a formal artifact — a dependency graph with declared contracts, validated by a compiler, frozen before execution, and replayable after success.
